# WolfCV :: Productization v0

## Status

Pencil document.

This is not a final product spec.
It is a working note for the phase change now happening in the project:

```text
WolfCV has already proven that the machine can work.
Now it must become usable by other people.
```

---

## 1. What changed

The old question was:

```text
can this machine exist at all?
```

That question has already been answered well enough.

The new question is:

```text
can another human use WolfCV
without sitting next to its author?
```

This is the real start of product work.

---

## 2. Product goal

`WolfCV` should become:

- understandable
- runnable
- inspectable
- trustworthy
- shareable

without losing its truth law.

It must not become a polished lie machine
just because it is easier to package.

So the product goal is not:

```text
make it look friendly at any cost
```

The real goal is:

```text
make the machine usable
while preserving evidence and falsification boundaries
```

---

## 3. Current state

What already exists:

- working truth contour
- working vacancy-aware contour on smaller surfaces
- GitHub ingestion
- vacancy diagnosis
- guard outputs
- multi-model substrate beginning
- machine-first documentation

What is still missing for real product use:

- one canonical user journey
- simpler external explanation
- clearer input expectations
- clearer output hierarchy
- visible degraded states
- more stable quality across different vacancy shapes

---

## 4. Product layers

The next product phase should be thought about in four layers.

### Layer A — Entry

The user must understand:

- what WolfCV is
- what to feed it
- what it will return
- what it will not promise

### Layer B — Experience

The user must be able to run:

- one obvious happy path
- with one obvious example
- and see one obvious result set

### Layer C — Trust

The user must be able to distinguish:

- strong result
- partial result
- degraded result
- unsupported claim
- genuine gap

### Layer D — Sharability

The project must be easy enough to hand to:

- a technical friend
- another strong repo-shaped candidate
- another machine reviewer

without live hand-holding.

---

## 5. Product law

`WolfCV` must become easier to use,
but not morally softer.

So productization must preserve:

- traceability
- bounded evidence
- guarded claims
- visible uncertainty
- explicit degraded states

If these disappear,
the product becomes just another AI helper.

---

## 6. Immediate product direction

Do not jump straight to UI cosmetics.

The correct near-term order is:

1. define canonical user flow
2. define canonical output surfaces
3. define trust surfaces
4. make the first external-facing docs
5. only then improve packaging and polish

---

## 7. What this phase is not

This phase is not yet:

- public launch
- brand design
- mass onboarding
- broad API productization

It is earlier.

It is:

```text
turn the machine into a tool another human can actually use
```

---

## 8. Short formula

```text
R&D proved the engine.
Productization must make the engine operable, legible, and trustworthy.
```

---

## 9. Optional later mode

Optional later product mode:

- `full hohotach`

Meaning:

- an explicitly enabled satirical vacancy-reading mode
- built on top of normal vacancy diagnosis
- never replacing the serious diagnosis layer

Why it may matter:

- pedagogical value
- anti-bullshit pattern recognition
- emotional relief when the market produces absurd vacancy texts

Important law:

```text
diagnosis first
roast second
```

This mode should remain optional and off by default.

---

## 10. Future runtime modes

`WolfCV` should not remain single-mode forever.

Future product/runtime modes should include:

- `full`
  - maximum depth
  - slowest
  - most expensive
  - canonical full compile
- `balanced`
  - normal everyday mode
  - reduced cost and latency
  - still serious enough for normal use
- `fast`
  - rough first pass
  - aggressive prefiltering
  - useful for quick repo/vacancy matching
- `vacancy_only`
  - read the vacancy without compiling repo truth
- `truth_only`
  - read repo truth without vacancy targeting
